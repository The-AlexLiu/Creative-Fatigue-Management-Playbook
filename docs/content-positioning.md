# 内容型 GitHub 项目定位

这个项目不是为了展示代码能力。

它更接近一个“认知仓库”：把广告投放里的判断方法、指标口径、复盘框架和实际操作模板沉淀下来，让别人能看出我如何拆问题、如何做判断、如何把经验变成可复用系统。

---

## 为什么放在 GitHub

GitHub 在这里不是代码托管工具，而是作品集容器。

相比一篇普通文章，GitHub 更适合承放这类内容：

| 内容类型 | 为什么适合放 GitHub |
|---|---|
| 方法论 | 可以长期迭代，不是一次性发布 |
| 模板 | 可以直接复用、复制、改造 |
| 数据字段 | 可以保留清晰版本和结构 |
| 判断规则 | 可以随着实践持续修正 |
| 作品集 | 可以展示思考方式，而不只是最终结论 |

这个项目真正展示的不是“我写了一个脚本”，而是：

> 我如何把素材衰退这个模糊问题，拆成一套可以被判断、执行、复盘和迭代的系统。

---

## 和代码项目的区别

代码项目通常回答：

> 这个工具怎么安装、怎么运行、能实现什么功能？

这个项目回答的是：

> 面对素材衰退，我会如何判断问题、设计指标、规避误判，并把经验沉淀成可复用工作流？

因此，项目里的脚本和 CSV 不是主角。

它们只是把方法论落到表格里的辅助物。真正的核心是：

- 问题定义；
- 指标口径；
- 判断顺序；
- 动作分层；
- 平台差异；
- 复盘方式；
- 常见误判。

---

## 读者应该怎么看

建议按这个顺序阅读：

1. 先看 `README.md`，理解整体判断框架；
2. 再看 `docs/methodology.md`，理解指标背后的逻辑；
3. 如果关心平台差异，再看 `docs/platform-notes.md`；
4. 如果想知道我如何修正原始思路，看 `docs/optimizer-review.md`；
5. 如果要落到日常表格，再看 `templates/` 和 `scripts/`。

不需要把它当成软件项目来读。

更合适的读法是把它当成一份投放方法论作品：

> 先看作者如何定义问题，再看作者如何拆指标，最后看这些判断如何落到日常运营动作。

---

## 这个项目要证明什么

这个项目希望证明几件事：

| 能力 | 体现方式 |
|---|---|
| 投放判断 | 能区分素材疲劳、受众饱和和漏斗问题 |
| 数据意识 | 不只看 ROAS，也看样本量、加权日龄、集中度和衰减趋势 |
| 运营系统化 | 能把素材管理从救火变成可复盘流程 |
| 平台理解 | 能区分 Meta、TikTok、Google Demand Gen / PMax 的操作差异 |
| 模板化能力 | 能把经验沉淀成字段、规则和可复用表格 |

这也是它和普通 GitHub 代码仓库最大的区别。

它不是为了展示“我能写什么功能”，而是为了展示“我如何思考增长问题，并把思考沉淀成可执行资产”。
